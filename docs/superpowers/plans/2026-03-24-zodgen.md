# @l4n3/zodgen Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a functional, pipe-composable Zod v4 fixture generator using `@faker-js/faker` for realistic test data.

**Architecture:** A layered system: public API → config transform pipeline → schema resolver → per-type generators → faker. The resolver walks the Zod schema tree via `_zod.def.type` discrimination, dispatches to constraint-aware generators, and supports user overrides via a first-match pipeline.

**Tech Stack:** TypeScript 7 (tsgo), Zod v4 (beta), @faker-js/faker 9+, Vitest, Biome, pnpm workspaces

**Spec:** `docs/superpowers/specs/2026-03-24-zodgen-design.md`

import type { Faker } from '@faker-js/faker'
import type { z } from 'zod/v4'
import type { CheckSet, GenContext, GeneratorConfig } from './types.js'

export const createCheckSet = (checks: ReadonlyArray<unknown>): CheckSet => ({
  // Use the actual field name for check discrimination (check vs kind)
  has: (name) => checks.some((c: any) => c._zod.def.check === name),
  find: (name) => {
    const found = checks.find((c: any) => c._zod.def.check === name)
    return found ? (found as any)._zod.def : undefined
  },
  all: () => checks.map((c: any) => (c as any)._zod.def),
})

export const createContext = (
  schema: z.ZodType,
  config: GeneratorConfig,
  path: ReadonlyArray<string>,
  depth: number,
  faker: Faker,
  generate: (schema: z.ZodType, key?: string) => unknown,
): GenContext => ({
  schema,
  def: (schema as any)._zod.def,
  path,
  depth,
  faker,
  config,
  checks: createCheckSet((schema as any)._zod.def.checks ?? []),
  generate,
})
```

**Note on `any`:** Zod v4 check types are not fully exported. The `any` casts are isolated to this factory. Adjust `c._zod.def.check` to the actual discriminator field.

import type { GenContext } from '../types.js'

export const generateString = (ctx: GenContext): string => {
  const { faker, checks } = ctx

  // Check format first
  // Use actual check names from Task 2 Step 1
  if (checks.has('email') || checks.find('string_format')?.format === 'email') {
    return faker.internet.email()
  }
  if (checks.has('uuid') || checks.find('string_format')?.format === 'uuid') {
    return faker.string.uuid()
  }
  if (checks.has('url') || checks.find('string_format')?.format === 'url') {
    return faker.internet.url()
  }
  if (checks.has('ip') || checks.find('string_format')?.format === 'ip') {
    return faker.internet.ip()
  }
  if (checks.has('cuid')) return faker.string.alphanumeric(25)
  if (checks.has('ulid')) return faker.string.alphanumeric(26).toUpperCase()

  // Length constraints
  const minCheck = checks.find('min_length')
  const maxCheck = checks.find('max_length')
  const min = minCheck?.value as number | undefined
  const max = maxCheck?.value as number | undefined

  // Starts with / ends with / includes
  const startsWithCheck = checks.find('starts_with')
  const endsWithCheck = checks.find('ends_with')
  const includesCheck = checks.find('includes')

  const prefix = (startsWithCheck?.value as string) ?? ''
  const suffix = (endsWithCheck?.value as string) ?? ''
  const include = (includesCheck?.value as string) ?? ''

  const fixedLength = prefix.length + include.length + suffix.length
  const targetLength = Math.max(min ?? 0, fixedLength + 1, 1)
  const maxLength = max ?? Math.max(targetLength, 20)
  const fillLength = Math.max(0, Math.min(maxLength, targetLength) - fixedLength)

  const fill = faker.string.alpha(fillLength)
  const result = `${prefix}${fill}${include}${suffix}`

  return max !== undefined ? result.slice(0, max) : result
}
```

**Note:** The format check detection pattern depends heavily on Task 2 findings. Zod v4 may use separate check types (`email`, `url`) or a single `string_format` check with a `format` field. Adapt accordingly.

export const generateNumber = (ctx: GenContext): number => {
  const { faker, checks } = ctx

  const isInt = checks.has('int')
  const multipleOf = checks.find('multiple_of')?.value as number | undefined

  // Determine bounds — adapt check names to actual Zod v4 names
  const gtCheck = checks.find('greater_than') ?? checks.find('gt')
  const gteCheck = checks.find('greater_than_or_equal') ?? checks.find('gte')
  const ltCheck = checks.find('less_than') ?? checks.find('lt')
  const lteCheck = checks.find('less_than_or_equal') ?? checks.find('lte')

  const hasPositive = checks.has('positive')
  const hasNegative = checks.has('negative')
  const hasNonnegative = checks.has('nonnegative')
  const hasNonpositive = checks.has('nonpositive')

  // Calculate effective min/max
  const rawMin = (gtCheck?.value as number | undefined) !== undefined
    ? (gtCheck.value as number) + (isInt ? 1 : 0.001)
    : (gteCheck?.value as number | undefined) ?? (hasPositive ? 1 : hasNonnegative ? 0 : -1000)

  const rawMax = (ltCheck?.value as number | undefined) !== undefined
    ? (ltCheck.value as number) - (isInt ? 1 : 0.001)
    : (lteCheck?.value as number | undefined) ?? (hasNegative ? -1 : hasNonpositive ? 0 : 1000)

  const min = Math.max(rawMin, hasPositive ? 1 : hasNonnegative ? 0 : rawMin)
  const max = Math.min(rawMax, hasNegative ? -1 : hasNonpositive ? 0 : rawMax)

  if (multipleOf !== undefined) {
    const minMult = Math.ceil(min / multipleOf)
    const maxMult = Math.floor(max / multipleOf)
    const mult = faker.number.int({ min: minMult, max: maxMult })
    return mult * multipleOf
  }

  if (isInt) {
    return faker.number.int({ min: Math.ceil(min), max: Math.floor(max) })
  }

  return faker.number.float({ min, max })

Create `packages/zodgen/src/resolve.ts`:

```typescript
import type { Faker } from '@faker-js/faker'
import type { z } from 'zod/v4'
import type { GeneratorConfig, OverrideMatcher } from './types.js'
import { createContext } from './context.js'
import { generators } from './generators/registry.js'

const matchOverride = (matcher: OverrideMatcher, path: ReadonlyArray<string>, ctx: ReturnType<typeof createContext>): boolean => {
  if (typeof matcher === 'string') return path.at(-1) === matcher
  return matcher(ctx)
}

export const resolve = (
  schema: z.ZodType,
  config: GeneratorConfig,
  path: ReadonlyArray<string>,
  depth: number,
  faker: Faker,
): unknown => {
  const generate = (childSchema: z.ZodType, childKey?: string): unknown =>
    resolve(
      childSchema,
      config,
      childKey !== undefined ? [...path, childKey] : path,
      depth + 1,
      faker,
    )

  const ctx = createContext(schema, config, path, depth, faker, generate)

  // Check overrides first (registration order, first match wins)
  const matchedOverride = config.overrides.find((o) => matchOverride(o.matcher, path, ctx))
  if (matchedOverride) return matchedOverride.generate(ctx)

  // Fall back to built-in generator
  const defType = (schema as any)._zod.def.type as string
  const generator = generators.get(defType)
  if (!generator) throw new Error(`No generator for type: ${defType}`)
  return generator(ctx)
}
```

**Important:** The `generate` function passed to context needs to propagate path segments. Object generator should call `ctx.generate(childSchema)` but the resolver needs the key name for path tracking. This requires the object generator to use a slightly different pattern — it should call a version of generate that accepts a key. Options:

1. Object generator accesses resolve directly (circular dependency)
2. The `generate` on context accepts an optional key parameter
3. Context has a separate `generateChild(key, schema)` method

**Recommended:** Option 2 — extend `generate` signature to `(schema: ZodType, key?: string) => unknown`. Update `GenContext.generate` type accordingly in `types.ts`.

- [ ] **Step 3: Implement fixture.ts**

Create `packages/zodgen/src/fixture.ts`:

```typescript
import { Faker, base, en } from '@faker-js/faker'
import type { z } from 'zod/v4'
import { applyTransforms } from './config.js'
import { resolve } from './resolve.js'
import type { FixtureGenerator, FixtureOptions, GeneratorConfig, Transform } from './types.js'

const createFaker = (seed: number | undefined): Faker => {
  const f = new Faker({ locale: [en, base] })
  if (seed !== undefined) f.seed(seed)
  return f
}

const generateOne = <T>(schema: z.ZodType<T>, config: GeneratorConfig, faker: Faker): T =>
  resolve(schema, config, [], 0, faker) as T

const generateMany = <T>(schema: z.ZodType<T>, count: number, config: GeneratorConfig, faker: Faker): T[] =>
  Array.from({ length: count }, () => resolve(schema, config, [], 0, faker) as T)

const defaultConfigInstance = applyTransforms([])

export const fixture = Object.assign(
  <T>(schema: z.ZodType<T>, opts?: FixtureOptions): T => {
    const f = createFaker(opts?.seed)
    return generateOne(schema, defaultConfigInstance, f)
  },
  {
    many: <T>(schema: z.ZodType<T>, count: number, opts?: FixtureOptions): T[] => {
      // Single faker instance shared across all items for seeded variety
      const f = createFaker(opts?.seed)
      return generateMany(schema, count, defaultConfigInstance, f)
    },
    create: (...transforms: ReadonlyArray<Transform>): FixtureGenerator => {
      const config = applyTransforms(transforms)
      return {
        one: <T>(schema: z.ZodType<T>): T => {
          const f = createFaker(config.seed)
          return generateOne(schema, config, f)
        },
        many: <T>(schema: z.ZodType<T>, count: number): T[] => {
          const f = createFaker(config.seed)
          return generateMany(schema, count, config, f)
        },
      }
    },
  },
)
```

**Note on `Object.assign`:** The spec's API design uses `fixture.many()` and `fixture.create()` dot notation. `Object.assign` is used for the top-level API; `fixture.create()` returns a plain `{ one, many }` object per spec.

## Parallelization Strategy

Tasks 1-4 are sequential (foundations).

Tasks 5, 6, 7 (generators) **can be parallelized across 3 agents**:
- **Agent A**: Task 5 — Primitive generators (string, number, boolean, date, bigint, literal, enum, nan, primitives)
- **Agent B**: Task 6 — Composite generators (object, array, tuple, set, map, record, template-literal)
- **Agent C**: Task 7 — Complex generators (union, intersection, nullable, optional, default, catch, lazy, branded, pipe, effects)

Each agent writes to its own generator files and its own test file (`generators-primitive.test.ts`, `generators-composite.test.ts`, `generators-complex.test.ts`) to avoid merge conflicts.

Tasks 8-11 are sequential (integration layer depends on all generators).
